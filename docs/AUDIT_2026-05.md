# NVR Appliance — аудит готовности к продакшену

**Дата:** 2026-05-12  
**Актуализация метаданных:** 2026-05-13  
**Версия кода:** `CPACK_PACKAGE_VERSION = 0.2.0`, OpenAPI **1.2.10**, UI `0.2.0`  
**Объём:** ~8 700+ строк C++/HPP, 32+ .tsx/.ts во фронте, unit-тесты при `NVR_BUILD_TESTS=ON` (см. [tests/](tests/)), GitHub Actions CI с **ctest** на Ubuntu 22.04/24.04, OpenAPI/Grafana/AppArmor/autoinstall.

## TL;DR

Архитектурный фундамент **готов на ~75–80 %**: правильный JWT-цикл, hardening systemd+AppArmor+nginx, шифрование при покое (libsodium ChaCha20-Poly1305), атомарная финализация сегментов с fsync, RAII-shutdown через `signalfd` и watchdog, очередь нотификаций с retry/backoff/dead-letter, аккуратное разграничение по ролям.

**Статус remediations (2026-05-13):** перечисленные ниже семь P0-пунктов **закрыты в текущем дереве** (`postinst`, `setup_ubuntu_nvr.sh`, `webui/package-lock.json`, kiosk URL, `nvr_cli`, тест архива, AppArmor). Исторические формулировки в разделе [P0](#p0--блокеры-продакшена) сохранены как контекст; ориентир для go/no-go — галочки в [чек-листе](#чек-лист-к-релизу) и блок [«Актуализация»](#актуализация-post-remediation-2026-05-13).

После регресса на железе проект пригоден к ограниченному production-pilot. Полный список замечаний ниже.

## P0 — блокеры продакшена

### P0-1. Setup-токен не читается демоном ✅

**Статус:** исправлено — `deploy/debian/postinst` выставляет `setup.token` в **`0640 root:nvr`** отдельным циклом после `0600` только для `initial-admin.password`.

`postinst` (актуально): сначала общий `0640` + `chgrp nvr` для секретов, затем **`0600` только для `initial-admin.password`**, затем снова **`0640` для `setup.token`**:

```23:42:deploy/debian/postinst
# Make sure secrets are readable only by root + nvr group when they exist.
for f in /etc/nvr-prototype/jwt.secret \
         /etc/nvr-prototype/master.key \
         /etc/nvr-prototype/initial-admin.password \
         /etc/nvr-prototype/setup.token ; do
    if [ -f "$f" ]; then
        chgrp nvr "$f" 2>/dev/null || true
        chmod 0640 "$f" 2>/dev/null || true
    fi
done
# initial-admin.password: root only (daemon must not read it).
# setup.token: daemon (user nvr, group nvr) must read it for /api/v1/setup/finalize.
for f in /etc/nvr-prototype/initial-admin.password; do
    [ -f "$f" ] && chmod 0600 "$f" || true
done
for f in /etc/nvr-prototype/setup.token; do
    if [ -f "$f" ]; then
        chgrp nvr "$f" 2>/dev/null || true
        chmod 0640 "$f" 2>/dev/null || true
    fi
done
```

Демон читает токен в `RouteApplication`:


### P1-7. Archive playback читает в память до 16 MiB на каждый Range ✅

Исторически буфер до 16 MiB жил в `HttpServer.cpp`; раздача сегментов теперь в `archiveSegmentMp4` (`src/api/RouteArchiveEvents.cpp`).

```49:71:src/api/RouteArchiveEvents.cpp
/// Max bytes read into RAM for one Range response (clients re-request further ranges).
constexpr size_t kArchiveRangeBodyMax = 4 * 1024 * 1024;
constexpr size_t kArchiveRangeChunk   = 64 * 1024;

std::string readFileRangeChunked(const fs::path& fp, size_t start, size_t len) {
    std::ifstream f(fp, std::ios::binary);
    if (!f) return {};
    f.seekg(static_cast<std::streamoff>(start));
    const size_t cap = std::min(len, kArchiveRangeBodyMax);
    std::string  out;
    out.reserve(cap);
    std::array<char, kArchiveRangeChunk> chunk{};
    size_t remain = cap;
    while (remain > 0) {
        const size_t n = std::min(kArchiveRangeChunk, remain);
        f.read(chunk.data(), static_cast<std::streamsize>(n));
        const size_t got = static_cast<size_t>(f.gcount());
        if (got == 0) break;
        out.append(chunk.data(), got);
        remain -= got;
    }
    return out;
}
```

**Статус (2026-05-13):** снижен пик RAM на один Range-ответ: потолок **4 MiB** на ответ, чтение с диска **64 KiB** за раз; **GET без `Range`** — `set_static_file_info_unsafe` (Crow отдаёт файл с диска без огромной строки `body`); **HEAD** с корректными `Content-Length` / `Content-Range`; при включённом **`X-NVR-Archive-Accel`** по-прежнему **X-Accel-Redirect** на nginx. Дальнейшее улучшение при необходимости — настоящий streaming body в Crow вместо накопления строки до 4 MiB.

## P2 — средний приоритет

### P2-1. autoinstall: хардкод пароля
`appliance/autoinstall/user-data:12` содержит литерал sha-512 placeholder. Сборка ISO без подстановки оставит пароль предсказуемым. **Фикс:** обязать `appliance/build_iso.sh` инжектить `mkpasswd -m sha-512` из env-переменной.

### P2-2. README ссылается на `/usr/local/bin`
`README.md:107`: «`sudo install -m0755 nvr-shell/target/release/nvr-shell /usr/local/bin/`». Конфликтует с принципом «`.deb` кладёт в `/usr/bin`».

### P2-3. Логи Python-хука растут вечно
`scripts/script.py` пишет в `/var/log/nvr-prototype/hooks.log` без rotation. `spdlog` ротейтит свои файлы, но хук работает в отдельном процессе/интерпретаторе. **Фикс:** logrotate-стенза.

### P2-4. Миграция 3 — смесь SQL и ad-hoc ALTER
```219:255:src/store/Database.cpp
if (!columnExists(*db_, "users", "totp_pending_enc"))
    db_->exec("ALTER TABLE users ADD COLUMN totp_pending_enc BLOB");
// ...и так далее 6 раз
```
Хрупко: если миграция упадёт между ALTER-ами, при повторном запуске часть таблиц уже мигрирована. Используйте `SAVEPOINT` или полный idempotent SQL.

### P2-5. nginx-конфиг не сегментирован по фичам
`deploy/nginx/nvr.conf` монолитен. Для подключения LE/self-signed надо подменять файлы. Хорошая практика — `conf.d/*.conf` + один корневой `server{}`.

### P2-6. Crow в `multithreaded()`-режиме
Crow's threading имеет открытые issue (race в websocket dispatch). Для прода надёжнее worker-pool с фиксированным числом potoков (Crow `concurrency()`) и SO_REUSEPORT-баланс через nginx.

### P2-7. JWT TTL хардкод
`Auth::issueSession` использует константы. Перенести в `nvr.yaml` (`auth.access_ttl_min`, `auth.refresh_ttl_days`).

### P2-8. `backup.pass` рутом и без ротации
Если файл утечёт — компрометируются все исторические бэкапы. Нужно: 1) ключи разные на каждый бэкап, 2) wrap-key хранится в HSM/TPM или у внешнего KMS. Минимально — добавить параметр `--rotate-pass`.

### P2-9. Setup-marker не имеет fsync/atomic-write ✅

**Статус:** запись через **`.tmp` → `fsync` файла и родительского каталога → `rename`**; при ошибке записи/переименования — **500** и удаление временного файла (`src/api/RouteApplication.cpp`, `/api/v1/setup/finalize`).

**Статус:** HTTP-код зависит только от **`PRAGMA quick_check`** БД; поле **`cameras`** в JSON — булев флаг «есть ли активные камеры», без влияния на 200/503 (см. `src/api/RouteInfra.cpp`). *(Историческое описание с `db_ok && has_cam` в аудите не соответствует текущему коду.)*

## Сильные стороны

1. **JWT lifecycle** — `jti` revocation, `token_version` bump на password-change/disable, rotating refresh — выше уровня многих open-source аналогов.
2. **`signalfd` + watchdog** в `main.cpp` — async-signal-safe shutdown и интеграция с `systemd Type=notify`.
3. **Атомарная финализация сегментов** — `.tmp` → `fsync` → `rename` → `fsync(dir)`. Power-cut устойчиво.
4. **Path-traversal hardening** — `pathInsideRoot` + `isRegularFileNoSymlink` повсюду, где идёт работа с пользовательскими/DB-путями.
5. **Notification worker-pool** — bounded queue, retry exp-backoff, DLQ, sanitize CRLF в Telegram/Webhook.
6. **HW-decoding fallback** — QSV → VAAPI → SW, без аварий.
7. **systemd hardening** — `ProtectSystem`, `SystemCallFilter`, `CapabilityBoundingSet` ровно на ту минималку, что нужна, плюс AppArmor сверху.
8. **TLS-offload на nginx** — даёт OCSP, HSTS, CSP, rate-limit zones (api/auth/hls), HLS-кэш — без переусложнения C++ части.
9. **Versioned migrations** — `schema_version`-табличка, заметные backward-сохранения (`columnExists` гард).
10. **UI** — `RequireRole` с серверным me-roundtrip, ErrorBoundary, lazy routes, локальная генерация QR через `qrcode` (без сторонних сервисов).
11. **Bootstrapping appliance ISO** — autoinstall заранее ставит deb + создаёт сервисного пользователя, late-commands валидируют sudoers через `visudo -cf`.
12. **CI/CD** — SBOM (anchore), build-provenance (attest-build-provenance), sha256 рядом с релизами.
