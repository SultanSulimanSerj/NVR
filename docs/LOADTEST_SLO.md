# Нагрузочные профили и SLO (k6)

Документ закрывает пакет **Parity p24**: как прогонять смоук **k6**, какие переменные окружения нужны и какие **ориентиры SLO** фиксировать на поле (матрица «каналы × разрешение»). Цифры ниже — **не жёсткие гарантии кода**, а шаблон для записи результатов прогона на вашем железе.

## Скрипты в репозитории

| Файл | Назначение | Авторизация | Переменные |
|------|------------|---------------|------------|
| [scripts/k6-nvr-smoke.js](../scripts/k6-nvr-smoke.js) | `GET /healthz`, `GET /api/v1/health` к демону напрямую (обычно loopback) | Нет | `BASE_URL` (по умолчанию `http://127.0.0.1:8080`) |
| [scripts/loadtest/k6-health.js](../scripts/loadtest/k6-health.js) | `GET /api/v1/health` через nginx/TLS | Нет | `NVR_BASE_URL` (по умолчанию `https://nvr.local`, без завершающего `/`) |
| [scripts/loadtest/k6-smoke.js](../scripts/loadtest/k6-smoke.js) | Логин, список камер, попытка `GET /live/{id}/index.m3u8` | Да | `NVR_BASE_URL`, `NVR_USER`, `NVR_PASS` |

Пороги в скриптах (пример):

- `k6-nvr-smoke.js`: `http_req_failed < 1%`, `http_req_duration p(95) < 500 ms` для перечисленных GET.
- `k6-health.js`: `http_req_failed < 10%` при 2 VU / 20 s (мягче — учёт сети до стенда).
- `k6-smoke.js`: `http_req_failed < 50%` на коротком прогоне (логин/HLS могут дать 404, если нет потока).

Подстройте `thresholds` в своей копии сценария под контракт со стендом.

## CI (GitHub Actions)

- **Сборка и `ctest`:** job `build` в [.github/workflows/ci.yml](../.github/workflows/ci.yml) на `ubuntu-22.04` и `ubuntu-24.04` (`cmake -DNVR_BUILD_TESTS=ON`, затем `ctest --output-on-failure`).
- **Playwright:** job `e2e` в том же workflow запускается **только если** в настройках репозитория задана переменная **`NVR_E2E_BASE_URL`** и секрет **`NVR_E2E_ADMIN_PASSWORD`** (см. [PRE_FIELD_AUDIT_2026-05-13.md](PRE_FIELD_AUDIT_2026-05-13.md) п.12).
- **k6:** workflow [.github/workflows/loadtest.yml](../.github/workflows/loadtest.yml): **workflow_dispatch**, шаг k6 только если задана переменная репозитория **`NVR_LOADTEST_BASE_URL`**. В job передаётся `NVR_BASE_URL` для `k6-health.js`.

## Матрица SLO для поля (заполнить при прогоне)

Запишите конфигурацию и **факт** после стабилизации (например 1–4 ч записи + live в UI):

| Профиль | Камер | Основной поток | Субпоток / motion | CPU средний / пик | RAM резидентная | Замечания (дропы, задержка live) |
|---------|------:|-----------------|---------------------|-------------------|-----------------|----------------------------------|
| A — офис | 4 | 1080p25 H.264 | 640×360 @ 8 fps | | | |
| B — магазин | 8 | 1080p25 | 640×360 | | | |
| C — стресс | 16 | 720p25 или 1080p15 | 640×360 | | | |

**Ориентиры для go/no-go (стартовые, уточнять на i7-class + QSV):**

- **CPU:** средняя загрузка всех ядер при **профиле B** не выше **~75%** в течение часа непрерывной записи + одновременного просмотра 2 live в UI; пик допустим кратковременно до **~95%** без watchdog-restart `nvr-prototype`.
- **Память:** рост RSS без утечки — плоская кривая после прогрева; **OOM** = no-go.
- **Диск:** запись без переполнения вне настроенной квоты; latency `statvfs`/eviction в пределах, не мешающих записи (субъективно — отсутствие «рваного» архива в UI).

Пороги декодера/пайплайна можно снять через `GET /api/v1/cameras/{id}/stream-stats` и журнал `nvr-prototype`.

## Как фиксировать результат

1. Дата, commit, модель CPU, RAM, тип диска архива.
2. Вставка строки из матрицы + цифры из `htop`/`pidstat`/Prometheus.
3. Ссылка на артефакт k6 (stdout) при автоматическом прогоне.
4. В чеклисте поля [PRE_FIELD_AUDIT_2026-05-13.md](PRE_FIELD_AUDIT_2026-05-13.md) отметить строку P2 про нагрузку и приложить ссылку на этот документ или отчёт.
