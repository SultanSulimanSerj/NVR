# MQTT-уведомления: статус продукта

В API (`/api/v1/system/info`, `/api/v1/system`) поле **`features.mqtt`** отражает наличие кода канала и базового контура; поле **`features.mqtt_delivery`** описывает фактическую доставку.

| `mqtt` | `mqtt_delivery` | Семантика |
|--------|-------------------|-----------|
| `true` | `"stub"`          | Канал в коде есть; брокер **не** подключён как продуктовая доставка. Тестовые сообщения по-прежнему могут попадать в DLQ. |
| `true` | `"live"` | Реальная публикация JSON в MQTT 3.1.1 QoS0 по TCP (`src/notify/MqttTcp311.cpp`), если в канале заданы `broker_host` / `broker_port` (или устаревшее поле `broker` как `host:port`). |

- **UI:** тип канала `mqtt` скрыт, пока `mqtt_delivery === "stub"`; после `PUT /api/v1/config/features` с `{"mqtt_delivery":"live"}` (или записи `features.mqtt_delivery` в SQLite) канал доступен.
- **Код:** `src/notify/MqttChannel.cpp` + минимальный TCP-клиент; TLS пока не поддерживается — только за reverse-proxy/VPN.
- **Контракт для интеграторов:** ориентироваться на **`mqtt_delivery`**, а не только на `mqtt`. Пока значение `stub`, считать доставку через MQTT **недоступной**; использовать email, Telegram и webhook.
- **Путь к включению:** выставить `mqtt_delivery: "live"` через `PUT /api/v1/config/features`, настроить канал `mqtt`, прогнать тест канала и DLQ.

Документ фиксирует продуктовое решение: **явная семантика флагов**, без расхождения с README и паритетными планами.

**Паритетный пакет p28:** контракт зафиксирован; переход на **`mqtt_delivery: live`** требует операторской проверки брокера и сети (без TLS на стороне NVR).
